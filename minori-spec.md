## 1. プロダクト概要

### 1.1 プロダクト名・コンセプト

**プロダクト名：みのり（MINORI）**

農家が複数の販路（JA・道の駅・ポケマル・食べチョク・ふるさと納税・飲食店直取引）を持つようになっている中、それらの売上・利益・在庫を一元管理するツールが存在しない。みのりはその空白を埋める農家専用のWebアプリSaaSである。

---

### 1.2 解決する課題

| 課題 | 詳細 |
|---|---|
| 販路がバラバラで管理できない | JA・道の駅・EC・ふるさと納税を別々に管理。「どの販路が最も利益が出るか」が分からない |
| 品質による価格変動が記録できない | 秀品・優品・規格外で単価が2〜3倍異なるが、既存ツールはグレード管理非対応 |
| 手数料が変動する（段階手数料） | 道の駅など月売上に応じて手数料率が変わる仕組みがある。固定率計算では利益がズレる |
| 確定申告・請求書が面倒 | 飲食店直取引の請求書作成・農業収支帳の作成が手作業 |

---

### 1.3 ターゲットユーザー

- **主ターゲット**：複数販路を持つ個人農家・家族経営農園（全国約180万人の個人農家の中でJA以外の販路を持つ層）
- **サブターゲット**：新規就農者・SNS発信して直販を始めた若手農家（30〜40代）
- **将来**：農業法人・複数農場を持つ経営者

## 2. 機能仕様

### 2.1 フェーズ別機能一覧

| フェーズ | 機能 | 概要 | 優先度 |
|---|---|---|---|
| MVP Phase 1 | 販路別売上ダッシュボード | 複数販路の売上を手動入力またはCSVで取り込み、グラフ・表で可視化 | **最高** |
| MVP Phase 1 | 品質グレード別売上入力 | 秀・優・良・規格外ごとの単価・数量を記録。グレード別利益率を自動計算 | **最高** |
| MVP Phase 1 | 段階手数料設定 | 販路ごとに固定率・段階手数料テーブルを設定。適用率と手取り利益をリアルタイム計算 | **最高** |
| Phase 2 | 在庫・出荷スケジュール管理 | 収穫量に対して各販路への出荷量を割り振り。二重注文防止・在庫警告 | 高 |
| Phase 2 | 請求書自動生成 | 飲食店・直取引先への請求書PDF出力。インボイス対応 | 高 |
| Phase 2 | 農業収支帳出力 | 確定申告用の農業収支帳を月次・年次で自動集計・CSV出力 | 高 |
| Phase 3 | AI販路最適化アドバイス | 蓄積データから「来月はポケマルを増やすと利益が+8%になります」と提案 | 中 |
| Phase 3 | SNS投稿自動生成 | 収穫記録・作業日報からInstagram/X投稿文をAIが自動生成 | 中 |
| Phase 3 | API自動連携（ポケマル等） | 各ECプラットフォームのAPIと連携して売上データを自動取得 | 低 |

---

### 2.2 各ページ詳細仕様

#### 2.2.1 ダッシュボード

| 項目 | 仕様 |
|---|---|
| KPIカード | 総売上・手取り利益・稼働販路数・最高利益率販路を4枚のカードで表示。期間切替（月・四半期・年）に対応 |
| 月次棒グラフ | 過去6ヶ月の販路別売上を積み上げ棒グラフで表示。各バーにホバーで金額ツールチップ表示 |
| 販路別内訳 | 今月の販路ごとの売上・比率をドーナツチャートと一覧で表示 |
| 作物別ランキング | 作物・グレード・販路・売上・利益率を表形式で表示。利益率をバーで可視化 |
| 最近の売上記録 | 直近5件の売上記録を日付・販路・作物・金額で表示 |

#### 2.2.2 売上入力

| 入力項目 | 仕様 |
|---|---|
| 販路 | JA出荷 / 道の駅 / ポケマル / 食べチョク / 飲食店直取引 / ふるさと納税 から選択 |
| 日付 | DatePickerで選択。デフォルトは当日 |
| 作物 | 事前登録した作物リストから選択（後から追加可能） |
| 品質グレード | 秀（最上位）/ 優 / 良 / 規格外 から選択 |
| 数量・単位 | 数値入力 + kg / 個 / 箱 / パック から単位選択 |
| 実際の単価 | 円/単位で入力 |
| 手数料タイプ | 固定率 / 段階手数料 を選択。段階の場合は売上に応じて自動で適用率を算出 |
| リアルタイム計算 | 入力中に「売上金額 → 手数料 → 手取り利益 → 利益率」をリアルタイムで計算・表示 |
| 段階手数料アラート | 「あと○円売ると手数料率が●%に下がります」を表示 |

#### 2.2.3 販路管理

- 各販路のカード表示：今月売上・手取り利益・利益率・手数料設定を一覧
- 段階手数料テーブル：売上額に応じた手数料率を複数段登録可能
- 販路比較テーブル：全販路の手数料額・手取り・利益率を横並び比較
- 新規販路の追加・編集・削除

#### 2.2.4 在庫管理

- 在庫一覧：作物ごとの在庫量・出荷予定量・残量・出荷先割振りを表示
- 在庫警告：在庫が出荷予定量を下回った場合に警告バナーを表示
- 出荷スケジュール：日付・販路・数量を登録して出荷予定を管理
- 収穫予定：将来の収穫量見込みを入力して在庫計画を立てる

#### 2.2.5 請求書

- 取引先情報（社名・住所・振込先）を事前登録
- 作物・グレード・数量・単価から請求書を自動生成
- インボイス登録番号（T番号）の表示に対応
- PDF出力機能
- 請求書番号の自動採番（年-月-連番形式）

## 3. データベース設計（Supabase / PostgreSQL）

### 3.1 テーブル一覧

| テーブル名 | 用途 | 主なカラム |
|---|---|---|
| users | 農家ユーザー | id, email, name, farm_name, location, created_at |
| routes | 販路マスタ | id, user_id, name, color, fee_type(fixed/tier), fee_rate, is_active |
| fee_tiers | 段階手数料テーブル | id, route_id, min_amount, max_amount, fee_rate |
| crops | 作物マスタ | id, user_id, name, unit（kg/個/箱/パック） |
| sales_records | 売上記録（中核） | id, user_id, route_id, crop_id, date, grade（S/A/B/X）, unit_price, quantity, total_amount, fee_rate, fee_amount, net_profit, memo |
| inventory | 在庫管理 | id, user_id, crop_id, stock_amount, updated_at |
| shipment_plans | 出荷スケジュール | id, user_id, route_id, crop_id, planned_date, planned_amount, status |
| invoices | 請求書 | id, user_id, client_name, client_address, issue_date, due_date, subtotal, tax, total, status |
| invoice_items | 請求書明細 | id, invoice_id, description, quantity, unit_price, amount |

### 3.2 重要カラムの補足（sales_records）

| カラム | 型 | 説明 |
|---|---|---|
| grade | VARCHAR | 秀=S、優=A、良=B、規格外=X。品質による単価差を分析するための核心カラム |
| unit_price | INTEGER | 実際に適用されたグレード別単価（円） |
| fee_rate | DECIMAL(5,4) | 実際に適用された手数料率。段階手数料の場合は売上に応じて変動した実際の値を保存 |
| fee_amount | INTEGER | 実際の手数料額（total_amount × fee_rate で計算） |
| net_profit | INTEGER | 手取り利益（total_amount - fee_amount）。利益率分析の基礎データ |

### 3.3 手数料設計

#### fee_basisカラム（routesテーブルに追加）

```
fee_basis: monthly | per_transaction | annual
reset_day: 月次リセット日（デフォルト：1日）
```

#### 計算パターン

| タイプ | 例 | 計算基準 |
|---|---|---|
| 固定率 | JA：売上の8% | 1回の売上に対して固定 |
| 段階手数料（月次） | 道の駅：月20万超で12%→15% | 月累計売上でリセット |
| 段階手数料（取引ごと） | ポケマル：1万円以上で15%→12% | 1回の注文ごとに独立計算 |
| 固定費＋手数料 | 食べチョク：月額3,000円＋10% | 月額固定費＋手数料率 |
| 年次累計 | JA奨励金：年間300万超で奨励金 | 年間累計でリセット |

## 4. 技術スタック

| レイヤー | 技術 | 選定理由 |
|---|---|---|
| フロントエンド | Next.js 14 App Router + TypeScript | 既存スキルをそのまま活用 |

| 認証 | Supabase Auth | メール認証・Google OAuth対応 |
| グラフ | Recharts | Next.jsとの相性が良い軽量チャートライブラリ |

| 精算書読取 | Claude Vision API | 道の駅・JA精算書をスキャンして自動入力（Phase 2） |

## 9. リスクと対応方針

| 課題 | リスク度 | 対応方針 |
|---|---|---|
| 競合の機能模倣（特にアグリノート） | 高 | 今のうちに農家との口コミ接点を増やして先行者優位を確保 |
| 個人情報漏洩 | 高 | Supabase RLSを最初から適切に設定。.envの管理徹底 |
| 農家のITリテラシーが低い | 中 | UIを極力シンプルに。初期設定は個別サポート（電話・LINE）で対応 |
| 有料転換率が低い | 中 | 段階手数料計算結果をぼかして表示→課金で解放する導線設計 |
| 一人開発が止まる | 中 | サーバーレス構成で放置しても動く設計。ドキュメント整備を並行 |
| 季節性チャーン | 中 | 年払い比率を上げる。農閑期も収支帳として使えることを訴求 |
| 外部サービス依存 | 低 | Prisma ORMでベンダーロックイン回避。移行手順を頭に入れておく |
| 制度変更対応（インボイス等） | 低 | freee連携でデータ出力する設計にして深追いしない |
